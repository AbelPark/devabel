캐러셀은 UI 개발 시 까다로운 녀석 중 하나이다.  
그러나 한정된 화면에서 컴팩트하게 컨테츠를 노출 하려면 거의 필수적인 요소라 이녀석과 마주하는건 피할 수 없는 운명이다.  
그래서 대부분의 회사에서 직접 구현하기 보단, 잘 만들어진 라이브러리를 사용, 적당히 커스텀하여 사용한다.  
나 또한 사내 프로젝트에 캐러셀을 적용해야 했다.  
일주일이 채 되지 않는 시간이여서(~사실 기간 많았어도 라이브러리 사용하려고 했음...~) 프로젝트에 알맞는 라이브러리를 찾아봤다.

구현하려는 캐러셀의 조건은

1. PC, Mobile에서 자연스럽게 잘 동작해야한다.
2. auto play 기능이 있어야 한다.
3. inspector 기능이 있어야 한다.
4. fade 효과가 있어야 한다,
5. 무료

2,3,4 번은 대부분의 라이브러리가 지원하는 기능이라 크게 고려하지 않았다.  
문제는 핵심인 1번,  
이 부분에서 라이브러리 선택 여부를 결정지었다.  
결론부터 말하면 전통의 강자 **swiper**를 선택했다.  
필요한 모든 기능을 간단하게 적용할 수 있고, 무엇보다 마우스로 터치해도, 손가락으로 터치해도 일관성있게 자연스럽게 동작했다.

일단 신뢰도 높은? npmtrends를 참고하여 다운로드 수 많은 대중적은 라이브러리를 살펴보았다.

양대산맥 siwper, react-slick를 픽 했고,  
추가로 떠오르는 다크호스(공식문서가 매우 훌륭함) embla-carousel을 살펴보았다.

지금부터 각각의 라이브러리의 체험기를 소개 하겠다.

## React-Slick

### 장점

- 기능 구현 간단함

### 단점

- 캐러셀 동작부에서 문제 발생

프로젝트 진행시 수시로 변경되는 기획, 디자인에 맞춰서 대응할 수 있는 간단한 기능구현이라는 훌륭한 장점에도 불구하고 slick을 사용하지 않은 이유는 동작부에서 치명적인 문제가 발생했기 때문이다.
캐러셀 전체 영역을 페이지 이동, 즉 `<Link>` 태그로 감싸줘야 했다.  
그런데, 이렇게 캐러셀의 DOM요소를 감싸주면 모바일에서는 괜찮았는데, 마우스로 드래그하여 넘길 때 간섭이 발생하여 제대로 동작하지 않았다.  
onClick event로 바꿔주면 동작은 하나, 간헐적으로 스와이프시 클릭 이벤트가 발생하곤 했다.
따라서 slick은 제쳐두고 다른 라이브러리를 찾았다.
